# 32 · 領域：QR 點名與 data token 逆向研究

QR 是四型點名中唯一沒有純學生端自動解的一型。本 doc 是這個問題的**完整研究紀錄與負面結果地圖**——
它是本專案的長期遺產，交給後繼者：**保留全部，別刪、別替換成敷衍結論。**

## 誠實三規矩（不可違反）

1. **只說「尚未發現」，不說「不可能」。** 純學生端偽造 QR `data` token 的方法**至今尚未被發現**——
   這是「還沒找到」，**不是**「不可能」。程式碼、文件、對外說明一律用「尚未發現」；**永不寫
   「不可偽造／不可能／impossible」**。
2. **不指名。** 若聽聞有人成功純學生端偽造，一律以「**傳聞／流傳／都市傳說**」稱之，**永不指名任何
   真實的人**。
3. **手動不偽裝成自動。** 任何手動貼上/掃描的輔助工具都必須誠實標為「手動、last-resort」，
   **永不包裝成「自動」或「預設支援」**。

## 目前唯一的自動路徑：教師輔助

用一個獨立的**教師 session** 自行起一場 QR 點名、讀出輪換的 `data`、代學生送出、在確認窗口內反覆刷新
重送、最後關掉教師端（見 `30-domain-rollcall.md`）。需配置教師帳號（能力旗標 `qr_teacher_assist`）。
這是**暫時的妥協**，不是解——它靠「持有教師身分」，而非「破解 token」。

## 已知的 token 性質

- QR 內含一個 `data` token，**隨時間輪換**。
- `data` **綁定時間 + 一個全域密鑰**：實機觀察到，教師端產生的 `data` 具**跨課、跨校可攜性**——同一批
  教師 `data` 能在不同課程、甚至不同學校（實測跨兩所）通過。這強烈指向「時間 + 全域密鑰」的簽章模型，
  而非 per-course/per-session 綁定。
- **生成祕密在伺服器端，不在 client。** 對廠商前端與端點的普查未在任何 client 側找到生成 `data` 的
  演算法或密鑰材料。

## 負面結果地圖（已實測排除的路）

以下方向都**實機試過、皆為負面**（＝此路目前不通，非「不可能」）：

- **密碼學結構分析**：對 `data` 的結構/編碼未還原出可自產的演算法。
- **洩漏路徑**：學生端各端點**吐不出 `data`**；曾經可能的洩漏點已死。
- **偽造/重放**：純學生端構造或重放 `data` 未能通過（教師端可攜是因其為合法簽章來源，非偽造）。
- **client 側複刻**：前端不含生成邏輯/密鑰，無法在 client 重現。
- **舊版本 API**：舊 `api_version`/舊端點未提供繞道。
- **IDOR / 越權**：對端點的角色×方法矩陣普查**未發現提權**能讓學生取得 `data` 或教師能力。
- **對稱加密假設（AES 等）**：以對稱金鑰重建的假設未獲驗證。

> 平台面普查另註：TronClass 線上 API 面積龐大（約 1852 端點），角色×方法矩陣**無提權**；`socket.io`
> **對學生無推播**（所以偵測只能靠輪詢，見 `30`）。基礎設施設定曾洩漏過憑證性資料——**此類洩漏物屬
> 機密，不納入本規格、不重製**；此處僅記錄「曾發現過設定面洩漏」這個事實。

## 真正的解會長什麼樣

要做到純學生端自動：**必須知道生成演算法 `f` 與全域密鑰 `KEY`**，讓學生端自算 `data = f(當下時間, KEY)`
得到有效 token。目前 `f` 與 `KEY` 皆未取得。

## 採樣方法論（給後繼者）

- **要打的是教師端的 `(server_time, data)` 對**——教師端的 `qr_code` 才吐得出 `data`；學生端吐不出。
  蒐集足量的時間戳 ↔ token 對，才有機會反推 `f`/`KEY` 的結構。
- **研究模式（research mode）只是蒐證儀器，不是解法。** 它 opt-in、限特定家族域、原文只落本機
  gitignored 位置、讀取時 re-redact，**絕不進任何匯出/分享包**。它負責把 `(t, data)` 對記錄下來供分析，
  本身不破解任何東西。
