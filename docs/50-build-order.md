# 50 · 建造順序（Build Order）

## 原則：設計全知、建造增量

全部功能集已知且已規格化（`00`–`40`），所以**架構可以一次切對**。但**不要在出貨前把全部功能都
實作完**。分片建造，**每片都落在已經對的架構上**，不用回頭改地基。這正是避免「邊做邊縫」的做法。

## 第 0 步：Walking skeleton（先證明最危險的三件事）

在寫任何功能之前，做一個**會走路的骨架**：

> 一顆按鈕 → Rust core 對一所真實學校做**一次真的登入** → UI 顯示結果。

這一個骨架必須**同時證明**：

1. **async 過 FFI**：C# 的 `await`/Task ↔ Rust 的 tokio future 能對起來。
2. **事件 callback**：core 能把非請求觸發的事件主動推上 UI（先推一個 `StateChanged` 就行）。
3. **平台把行程吊活**：桌面天生活；Android 起前景服務；iOS 確認前景可跑。

這三點是整個專案的成敗關鍵（見 `10-architecture.md`）。**skeleton 綠了才往下。** 別在這步塞任何領域功能。

## 之後的分片（每片獨立可跑、落在同一架構上）

順序可調，但大致由「地基與成熟功能」往「複雜與邊緣」推：

1. **登入 + 帳號**：登入特徵分流（`30`）、帳號 CRUD、切換、session cookie 快取；`SecretStore` 起步版
   （加密庫 + 主密碼，見 `10`）。
2. **監控迴圈 + 成熟點名**：輪詢分型（`30`）、去重、自適應輪詢；接 **number 與 radar**（最成熟兩型）；
   接上 **15% 門檻 + 15 秒倒數 + defer** 的點名有時限流程（`20` 流程 B）。
3. **答題**：exam 取題/決策/送出（`31`）；**衝突高亮 + 暫緩/捨棄 + reasoning stream** 的答題有時限
   流程（`20` 流程 A）；replay + resubmit 保滿分；LLM 客戶端（reasoning 常開、必送 max_tokens、
   工具讀教材、多模態）。
4. **QR 教師輔助**（`30`/`32`）：需教師帳號；能力旗標 `qr_teacher_assist`。
5. **self_registration**（`30`）。
6. **設定 UI + 加密庫完整版**：結構化設定頁；平台密鑰庫/生物辨識解鎖層（`10`）。
7. **平台保活與通知**：Android 前景服務、桌面常駐、本機通知（把 `20` 的倒數/事件接成通知）；
   自動更新（桌面/APK；iOS 不做，旗標關）。
8. **延後**：投票/其他答題來源、群組、聊天機器人通知、研究模式儀器（`32`）、瀏覽器手動登入後備、
   OCR 驗證碼 addon。這些不進 v1 的關鍵路徑。

## 每片的完成標準

- **core 有離線測試**：對假 TronClass 伺服器，用命令驅動、斷言事件（見 `10`/`90`）。
- **UI 能把該片的畫面走通**（職責級即可，視覺你設計）。
- 不引入不必要的依賴/抽象（`90` 的 YAGNI）。
- 若某片刻意留了捷徑/上限，在碼裡標明上限與升級路徑。
